4.2 Cache Memory Usage Increased

As mentioned in the primary readme, the DBMS server cache size defaults have been increased in Ingres 2006. As a result, the default DBMS vms_page_file parameter may not be sufficient when upgrading from 2.0 or 2.6, particularly if you have enabled several caches for different page sizes. The default value for vms_page_file is 120,000 in Ingres 2.0, 250,000 in Ingres 2.6, and 2 billion in Ingres 2006. Before upgrading, we recommend that you increase the parameter using Configuration-By-Forms, DBMS Server, Configure, Derived.

5.3 Kerberos Service Principal Host Name Resolution

The Kerberos Key Distribution Center (KDC) will not resolve the fully qualified host name (FQDN) correctly (even though you specify it on the Configuration-By-Forms domain parameter) unless it resolves the host name passed from the client as the FQDN.

The FQDN is picked up from your network configuration (rather than the Ingres config.dat setting) when the Kerberos driver calls gss.init.sec..context(). Often the unqualified host name is passed to the KDC, and gss.init.sec..context() fails.

To ensure that the KDC can resolve the fully qualified host name

Edit the local host file with the FQDN and not the alias for your local host as the first entry.

Use the TCPIP utility:

$ tcpip

 

TCPIP> show hosts

 

      LOCAL database

 

Host address    Host name

 

127.0.0.1       localhost

nn.nn.nn.nn     myhost.mydomain.com   myhost

For details about configuration of the local hosts file, see your system administrator.

Kerberos commands are accepted in uppercase by default (as in DCL commands). Accordingly, principal names should be enclosed in double quotes, as in the following example:

$ kinit "tingresxx"

If "tingresxx" had not been enclosed in double quotes, the KDC would have looked for the principal "TINGRESXX" instead of "tingresxx". Creation of the Kerberos principal "TINGRESXX" (all caps) will not help; the Ingres security mechanism fails the authentication due to "TINGRESXX" not equating to "tingresxx". The Ingres security mechanism is still in force as the default authentication mechanism regardless of whether Kerberos is configured.

If server-to-server authentication is desired (CBF, Security, Configure, user_mechanism), the VMS logical KRB$USER must be defined at the group level. The SYS$MANAGER:KRB$SYMBOLS.COM script defines only at the process level by default. Otherwise, security contexts will not be accepted and the error "GSS-API error gss_init_sec_context: Credentials cache I/O operation failed XXX" will appear in a GCS trace or the error log. This problem occurs because servers run as detached processes. Detached processes run without a CLI and have no visibility of logical definitions at the process level.

The following command file defines KRB$USER at the group level based on the process-level definition, runs ingstart, and de-assigns KRB$USER:

$!

$! RINGSTART.COM - Run Ingstart with KRB$USER

$!

$ krb = F$TRNLNM("KRB$USER","LNM$PROCESS")

$ define/group/executive/translation=concealed KRB$USER 'krb'

$ ingstart

$ deassign/group/exec KRB$USER

$ exit

5.5 Lock Limits

In a cluster, or if you use Distributed Multi-Cache Management (DMCM) protocol without a cluster, the default lock limit may be insufficient. DMCM requires page cache locks that persist beyond a given transaction. As a result, you may want to increase rcp.lock.lock_limit to account for the total number of cache pages defined for the node.

6.0 Upgrading Cluster Installations

6.1 Upgrading Cluster Installations from Ingres 9.1.1

To upgrade a cluster installation from Ingres 9.1.1, you must first undo the existing cluster configuration.

To upgrade from Ingres 9.1.1 cluster installation

1.    Back up your current installation databases.

2.    Run iiuncluster.

The cluster Ingres instance is converted to a non-cluster (stand-alone) Ingres instance.

3.    Install Ingres 10.1 in stand-alone mode.

4.    Run iimkcluster on the master node, and then run iisunode on the other cluster nodes. For full instructions for converting a standalone installation to clustered, see the Installation Guide.

All nodes in the cluster installation are configured.

6.2 Upgrading Cluster Installations from Ingres 2.0

It is not possible to do a direct upgrade of an Ingres 2.0 cluster installation to Ingres 10.0. Instead, to safely upgrade from Ingres 2.0, use the procedure in this section.

If you want to automate this upgrade procedure, a saveset is available for download on the community web site.

To upgrade from Ingres 2.0 cluster installation

1.    Shut down Ingres 2.0 on every node in the cluster.

2.    Reconfigure to a standalone installation by doing the following:

a.     Edit config.dat and delete all the lines that contain the following:

●       ii.node1.config.cluster.id

●       ii.node1.config.cluster_mode

●       ii.node1.gcn.cluster_mode

where node1 is the lowest numbered node.

b.    Delete all config.dat entries for the other nodes.

For example, in a three-node cluster with machines named alpha, beta, and gamma, where alpha is the lowest numbered node, delete:

●       ii.alpha.config.cluster.id

●       ii.alpha.config.cluster_mode

●       ii.alpha.gcn.cluster_mode

●       ii.beta.* (all entries)

●       ii.gamma.* (all entries)

c.     To verify that the reconfiguration to standalone mode completed successfully, first edit the transaction log file logical name line in config.dat as follows:

ii.node1.lnm.ii_log_file_name:     ingres_log.system_

to

ii.node1.lnm.ii_log_file_name:     ingres_log.system_node1

Second, run ingstart, verify the operation, and then run ingstop.

Finally, delete the transaction log file logical name line (as follows) from config.dat:

ii.node1.lnm.ii_log_file_name

If you do not want to verify standalone mode operation, you can simply delete the previous line.

d.    If necessary, change ii.*.config.server_host to the value for node1.

3.    Use CBF (or a text editor) to edit ii.node1.dbms.*.vms_page_file to up to 2 billion (or at least 250,000). See Cache Memory Usage Increased in this readme.

4.    (Optional) Perform cleanup.

a.     To prevent later problems with installed images, we recommend that you run the following from each node:

@II_SYSTEM:[ingres.utility]iishare delete

b.    To prevent problems with transaction log files, you can also delete the files and the logicals. For example:

$ delete II_LOG_FILE:[ingres.log]ingres_log.system_*;*

$ deassign /group /executive_mode II_LOG_FILE

$ deassign /group /executive_mode II_LOG_FILE_NAME

The de-assignments must be done on each node. The above example presumes GROUP level logicals and should be adjusted accordingly for a system-wide installation.

c.     Before beginning the installation of Ingres 10.0, you may want to delete any job level logical by running the following:

@II_SYSTEM:[ingres]iijobdel

5.    Install Ingres 10.1 on one of the cluster nodes, typically the one with the lowest numbered cluster ID.

6.    Run iimkcluster and iisunode to reconfigure your cluster. Full instructions for converting a standalone installation to clustered are provided in the Installation Guide.
